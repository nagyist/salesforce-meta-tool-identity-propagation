# Prompt Agents: Context Isolation and Control Layers

Guidance for Azure AI Foundry **prompt agents** (portal-defined, server-managed) when you need stronger isolation, runtime controls, and slimmer context without duplicating the agent definition or shipping custom code.

## 1) Sub-Agent Pattern for Context Isolation
- **Recommended pattern today:** Expose a single HTTP tool in the main agent that calls an APIM `/delegate` endpoint. The gateway starts a brand-new run/thread against a dedicated *delegate* prompt agent (or code-first agent if allowed), passes only the scoped task (e.g., *discover schema for object X*), and forces the delegate to respond with a compact summary or a handle to cached raw data. Only that summary/handle is returned to the parent thread, keeping exploration noise out of the main context.
- **Keep identities straight:** Run the delegate under the same connection the parent uses when the result must respect per-user permissions; otherwise run under a service identity and make that explicit in the returned summary.
- **Self-reference risk:** If the parent can call itself, you can create infinite recursion, token blow-ups, duplicate side effects, and confusing audit trails. Do not register the parent agent as a callable tool. Use APIM to reject calls where `x-parent-agent-id` equals `x-target-agent-id`.
- **Recursion guard:** Add a depth header (for example `x-agent-delegate-depth`) in APIM; increment on every `/delegate` call and reject when `>1` (or your chosen limit). Also enforce a max-hop TTL in any cache that stores intermediate results so orphaned handles expire quickly.

## 2) Behavioral Control Without Editing the Agent
- **Prompt decoration via APIM:** `set-body`/`set-variable` can prepend a decorator system message (e.g., *"Delegation disabled for this request"* or *"Read-only mode"*) before forwarding to Foundry. This is viable so long as you preserve the schema of the Responses API payload and re-run after service version bumps to catch shape changes.
- **Runtime tool stripping:** You can remove tool definitions from the outbound JSON in APIM to disable specific tools per request (e.g., strip `write_record`). If you do, also rewrite `tool_choice` to `auto` (or a safe allowlist) to avoid the service rejecting unknown tool names. Expect the agent to re-plan; calls to stripped tools will fail fast.
- **Failure modes:** Schema drift in the service can break brittle JSON rewrites; malformed bodies will return 400s before reaching the agent. Stripping tools that the conversation history references can produce confusing model behavior. Debugging is harder because the request observed by Foundry differs from the client’s original intent—log both the inbound and rewritten payloads.

## 3) Context Compaction for Prompt Agents
- **Native knobs:** Portal prompt agents do not currently expose explicit compaction controls. The service enforces model-window truncation but keeps tool inputs/results verbatim, so threads grow quickly.
- **Agent Framework compaction SDK:** `TruncationStrategy`, `SlidingWindowStrategy`, and `SummarizationStrategy` in `azure-ai-agents` apply to code-first/in-memory agents. They are not plug-and-play for portal-managed prompt agents.
- **Run-level truncation flags:** `truncation_strategy`/`max_prompt_tokens` settings on a run request are not forwarded into Connected Agent calls and are not available for prompt agents created in the portal. Treat compaction as a gateway responsibility.
- **Practical patterns:** (1) In APIM, collapse older tool call results into a short summary and replace the original payload before it hits Foundry. (2) Return large tool outputs as handles (cache key + checksum) and let the main agent fetch a summarized view on demand; expire the cache aggressively. (3) For long threads, preface each request with a rolling summary message generated by a lightweight summarizer agent, and drop stale turns from the forwarded payload.

## 4) Meta-Tool Pattern and Ephemeral Lookup Results
- **Make lookups disposable:** Route schema/object discovery through the delegate path so the raw field catalogs live only in the delegate thread. Return a compressed summary (top fields, key flags, example query) to the parent and discard the delegate thread afterward.
- **Where to compress:** Best options are (a) inside the tool implementation (if you own it), (b) in APIM right after the tool response, or (c) via a tiny summarizer agent that rewrites the tool result before it re-enters the parent thread. Aim for fixed budgets (e.g., 1–2 KB) instead of proportional to schema size.
- **Replace rather than append:** After the agent consumes a lookup, overwrite the corresponding message in the forwarded payload with a summary + handle ID. Keep the raw payload in a short-lived cache in case the agent asks to “see details” within the next few turns.
- **RAG vs. meta-tool inflation:** Both add bulk, but RAG chunks are usually pre-sized and compressible, while schema dumps can explode unpredictably. Apply RAG-like mitigations (chunking, summaries, handles), but be stricter about TTLs and size caps because schema payloads tend to dominate threads faster.
